# Project Philosophy

JARVIS is an Operating Intelligence.

It is NOT a chatbot.

It is NOT a single AI model.

It is a modular operating system powered by AI.

Every module has one responsibility.

The architecture is more important than adding features.

---

# Primary Objective

Improve the project while preserving architecture.

Never destroy existing systems.

Never rewrite working code.

Always perform minimal modifications.

---

# Development Workflow

Step 1

Read documentation.

Step 2

Understand architecture.

Step 3

Inspect existing implementation.

Step 4

Locate the correct module.

Step 5

Modify only required files.

Step 6

Verify compatibility.

Step 7

Update documentation if architecture changes.

---

# File Modification Rules

Always mention

Files Modified

Files Created

Files Left Untouched

Never modify unrelated modules.

# Architecture Rules

Kernel coordinates.

Brain thinks.

Memory stores.

Voice speaks.

Automation executes.

HUD displays.

Vision observes.

Agents plan.

Never mix responsibilities.

---

# Import Rules

Never invent imports.

Never invent folders.

Never invent filenames.

Always verify existing project structure before importing.

---

# Code Generation Rules

Never rewrite an entire file unless explicitly requested.

Prefer minimal patches.

Preserve comments.

Preserve formatting.

Preserve compatibility.

---

# Error Handling Rules

If a required module is missing

STOP.

Report the missing dependency.

Do not fabricate replacement code.

---

# Documentation Rules

Whenever architecture changes,

update

PROJECT_ARCHITECTURE.md

CURRENT_STATUS.md

CHANGELOG.md

ROADMAP.md (if necessary)

---

# Git Rules

Never modify Git history.

Never force push.

Never rewrite commits.

Every logical feature should have its own commit.

Recommended Commit Format

Feature

Fix

Refactor

Docs

Performance

Architecture

Examples

Feature: Added startup audio

Fix: Corrected kernel imports

Docs: Updated API reference

Refactor: Simplified voice queue

---

# Prompting Rules

When responding,

always provide

Reasoning

Files to modify

Implementation steps

Compatibility impact

Testing recommendations

Never provide unexplained code.

# Configuration Rules

Never hardcode

API Keys

Voice Settings

Device Names

Timeouts

Volumes

Wake Words

Paths

jarvis_config.json

or

utils/config.py

# AI Assistant Behavior

You are contributing to an existing architecture.

You are not redesigning it.

Respect existing modules.

Respect existing file structure.

Respect existing coding standards.

Do not create unnecessary files.

Do not duplicate logic.

Do not rename stable modules.

---

# If You Are Unsure

Do NOT guess.

Do NOT fabricate.

Ask for the required file.

Or clearly state which implementation is missing.

Maintainability Always.
